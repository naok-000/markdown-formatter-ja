# 0006. SPEC.md, TODO.md, ADRの役割を分ける

## Status

Accepted

## Context

markdown-formatter-ja では，TDDで小さく実装を進める．
そのため，実装中に思いついた作業，後で検討する案，現在のproject方針，過去の判断理由を同じ文書に混ぜると，判断基準が読みづらくなる．

特に `TODO.md` は，Kent Beckやt_wadaのTDDで使うTODO listのように，実装中の短期記憶を外へ出すために使いたい．
一方で，`SPEC.md` は実装中に迷ったときに戻る判断基準として保ちたい．

## Decision

`SPEC.md` は，現在のproject目的，方針，外部から見える重要な仕様を書く文書とする．
実装詳細，細かい設計，過去の経緯は原則として書かない．
目的や方針が変わった場合は，古い内容を残さず，現在の判断基準として更新する．

`TODO.md` は，TDD中の作業メモとして使う．
実装の最小単位，次に試すtest，実装中に思いついたが今すぐやらない案を書く．
完了したTODOは適当なタイミングで削除してよい．
`TODO.md` はsource codeとの完全な同期を求めない．

設計判断の理由，捨てた案，方針変更の背景を残したい場合はADRに書く．
`SPEC.md` から詳細な判断理由へ誘導する必要がある場合は，ADRへのlinkだけを置く．

`SPEC.md` とADRはgit管理する．
`TODO.md` は作業メモとして扱い，git管理上の正確性や履歴を要求しない．

## Consequences

- `SPEC.md` を現在の判断基準として短く保ちやすい
- TDD中に思いついた案を `TODO.md` へ退避し，すぐ実装しない選択をしやすい
- 完了済みTODOや古い方針によるcontext増加を避けやすい
- 判断理由が必要な場合はADRに残せる
- `TODO.md` は共有された正確な進捗表としては扱わない
